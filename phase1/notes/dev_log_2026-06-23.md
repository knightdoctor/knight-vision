## 2. The real story: two underlying bugs in depth.py

The first cloth + B1 attempt this morning produced 0 chest box points,
just like the failed runs yesterday. I added the cloth, verified by
eye that the L_rect chest patch had std 4.23 (vs control 0.5) — real
fabric weave, exactly what Phil asked for — and SGBM *still* returned
NaN at the chest pixel.

Searching pre-rot R_rect row 640 for any textured patch surfaced the
chest at u≈680 with std ~4. The chest in pre-rot Cam A's L_rect sits
at u=399. So u_L − u_R = −282 — **negative disparity**. SGBM with
`minDisparity=0` only searches positive disparities. The chest was
unmatched even when the texture was visible because the geometry put
Cam B in the role of the "left" camera.

### Fix A — cam-1 auto-selection

Added a chest-projection probe before `stereoRectify`: project the
chest centre through each pre-rotated cam's intrinsics, compare u
coordinates, pick whichever has the LARGER u as cam-1 (the SGBM
"left"). Swap source-image roles and the world-frame origin for the
3D back-projection accordingly. With this fix in place, B1 produced
9000+ valid pixels per frame and the cloud reached the chest. But the
cloud landed too HIGH (shared-Y up to 0.26, chest box upper bound
0.23) — the pitched cameras only see the upper hemisphere of the
torso. **Smoke test box widened: half_y 0.05 → 0.10.**

### Fix B — the deeper bug: 3D rotation matrix for CW image rotation
   was inverted

Validating fix A by tracing a chest world point through the chain
(world → Cam B old frame → pre-rot → rectified → back-projected) gave
P_world = (0, 0, 0.005) — far from the actual chest at (0, 0.10,
0.18). The chain wasn't self-consistent.

Re-derived `R_OLD_TO_NEW` from the pixel-level CW rotation rule
(col_old, row_old) → (h_old − 1 − row_old, col_old). The matrix in
`depth.py` was the transpose of the correct one — `_R_OLD_TO_NEW_CWROT`
and `_R_NEW_TO_OLD_CWROT` had their roles silently swapped.

Why Week-1 still worked with the wrong matrix: in Week-1's convergent
look-at geometry, the chest sat exactly on Cam A's optical axis
(P_cam = (0, 0, Z), X = Y = 0). Both versions of the matrix produce
identical results when X = Y = 0, because they differ only in the
first two columns. The bug was invisible until the parallel-axis B1
put Cam B's chest off-axis (P_B = (0, −0.101, 0.5)).

Fix: invert `_R_OLD_TO_NEW_CWROT`'s definition. Verified by round-trip:
the chest in pre-rot Cam B (col=623, row=640, depth 0.5 m) now
back-projects to world (0, 0.10, 0.18) ✓.

This was the **same bug** that produced yesterday's "scene-texture"
diagnosis. The variants did differ, but the disparity-sign
incompatibility meant cloth's added texture couldn't help. Fix B
unblocked every variant, not just cloth.

## 4. Scene + matchability physics — confirmed

Cloth (fabric noise scale ~10 cm cycle, contrast 0.30, bump 0.10):
- Raw L chest patch std 4.70 (control: 2.22)
- Pre-rot L_rect chest patch std 4.23 (control: 0.5)
- Wholly enough for SGBM to find matches even with `uniquenessRatio=10`

Projector (overhead emission plane, scale 50 generated-coord, dot
threshold 0.70):
- Pattern *is* visible in raw frames (the plane emits speckle)
- BUT the plane physically blocks the existing IR-LED and SUN
  illumination → chest in raw image is dark (mean 48 vs 230 for
  cloth/control)
- Eevee doesn't fully support textured-emission lights without
  switching to Cycles; deferred

This means: for the foreseeable simulation work, **cloth (or any
textured-surface variant) is the path**. Active projection is a
hardware fallback for real scenes where the subject's surface is
genuinely flat-textured; in simulation we can just texture the surface.

## 5. Considered and rejected

(a) **Cranking SGBM `uniquenessRatio=0`** — yesterday's fallback when
the chest wasn't matching with bare skin. Reduces SGBM rigour, lets
spurious matches through. Not a real fix — masking the bug B above.

(b) **Wider FOV (90°/110°)** for the spec L-overhang convergent
geometry — yesterday's attempt. The spec geometry has its own R1
problem (cx_rect goes off-image), and cranking FOV makes it worse.
B1's parallel-axis simply avoids that whole failure mode.

(c) **`projector_emission_plane`** — works as a Blender object but
behaves like an occluding plane in Eevee, blocking the IR LED. To
make it genuinely additive would mean Cycles + shadow visibility off
on the plane; not worth the SGBM-cost trade-off when cloth already
solves the matchability problem.

(d) **Strict world-vertical baseline for B1** — was Phil's initial
specification. The 30° pitch + world-vertical baseline produces a Tz
component in the relative-pose pre-rotation that forces R1 to rotate
chest off-image. The cam-local-Y baseline (perpendicular to the
shared optical axis, in the pitch plane) is the only direction that
keeps R1 ≈ I. Physically: both cams on the same rigid mount,
separated perpendicular to the lens axis — exactly how stereo rigs
are actually built.
