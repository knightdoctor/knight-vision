## 1. Polar BLE stall (Jetson side)

**Symptom.** Standard HR Service notifications arrived for ~15-30 s then
silently froze. Device stayed `Connected: yes` per `bluetoothctl`; bleak
saw no errors; HR/RRI buffers stopped growing. Daemon kept POSTing the
last-known values to the viewer indefinitely, looking healthy from the
outside.

**First fix tried.** Increase RRI buffer maxlen. Useless — the buffer
was already at 300; the problem was upstream, no new notifications were
arriving to buffer.

**Actually fixed by.** Stall watchdog in the BLE loop: track wall time
of last RRI, if > 6 s without one, break the inner async-with and
reconnect. Wrapped the connect in an outer `while self._running` retry
loop with 2 s backoff. Disconnect throws `org.freedesktop.DBus.Error.
UnknownObject "Method Disconnect"` on reconnect — benign, swallowed.

**Considered and rejected.** Polar PMD service for accelerometer-based
RR. The PMD service IS present on H10 1774F932, but bleak 3.0.2 +
bluez 5.64 on Jetson hung at `start_notify` on the `indicate`
characteristic (PMD Control Point). Service discovery races compounded
the flakiness. Concluded the Linux/bluez stack itself was the blocker,
not the algorithm, and moved Polar to the Mac (CoreBluetooth — stable).

**Status.** Watchdog works on Jetson for the standard HR Service.
Mac bridge is the production path for now; Jetson Polar daemon kept as
fallback only.

## 2. Run #3 "harmonic-locking" investigation

**Symptom.** Run #3 (this morning's clean-bg run) per-window BPMs walked
21 → 21 → 23 → 24 → 25 → 26 → 38 → 100 → 118. The 20-26 BPM cluster
was suspected to be a 2×-fundamental lock against a true ~10 BPM
breathing rate.

**First hypothesis.** Algorithm picks 2× harmonic when fundamental is
weaker — same failure mode as ECG-RR pipelines. Spec'd PR5
"prefer lowest in-band peak above noise threshold".

**Actually the answer (harmonic_diagnostic.py).** Dumped spectra at
t = 10/30/60s. Found **no ~10 BPM peak at any time slice** — neither the
fundamental nor its harmonic. The dominant 23-26 BPM peak was a
real persistent signal in the whole-subject centroid, NOT a respiratory
harmonic. By t = 60s the dominant peaks were 65 BPM (HR fundamental) and
117 BPM (HR 2nd harmonic) — cardiac ballistocardiography, not breathing.

**Real fix (PR E + PR I).** Whole-subject centroid was the wrong signal.
Switched to chest sub-region centroid (PR E) + power-weighted band
centroid instead of single-peak-pick (PR I). After PR I, Run 4 (paired
with Polar) gave LiDAR 14.6 BPM vs Polar 14.1 BPM. The "lowest in-band
peak" fix would not have helped — there was no low peak to prefer.

**Considered and rejected.** (a) PR5 band-confidence weighting. The
diagnostic showed it would have weighted a non-existent peak. (b)
"Trailing window settled-median". Algorithmically right but masked
Run #3's first 7 settled windows at 8-10 BPM. Logged PR G as a follow-up
to switch to longest-anywhere settled subsequence.

## 3. Confidence-label miscalibration

**Symptom.** Pipeline was labelling HIGH at SNR 4.6 and MEDIUM at 1.7 —
not clinically defensible. Original thresholds 3.0 / 1.5.

**First action.** Tightened to HIGH ≥ 5, MEDIUM ≥ 3, LOW < 3 (PR A).
This was the right move for peak-pick SNR.

**But then PR I changed the underlying signal.** Power-weighted centroid
integrates across the in-band power distribution. SNR there is still
defined as `peak_in_band / mean_in_band` — but with the band tightened
to 10-30 BPM and the centroid averaging across a wider distribution,
peak/mean ratios are systematically lower than they were with the wide
band + peak-pick. Run 4 came out 14.6 BPM "MEDIUM" SNR 4.5 — the BPM is
clinically correct (Δ vs Polar = 0.5 BPM), but the label undersells it.

**Decision.** Don't recalibrate thresholds by guessing. Wait for N≥5
paired runs against Polar, plot confidence-label distributions against
|LiDAR − Polar| error, set thresholds empirically. This goes on the
next-session list, NOT into a 9pm config tweak.

**Considered and rejected.** Adding a separate centroid-mode SNR metric
(e.g. "band-power concentration"). Plausible but a second metric to
explain — keep peak/mean for now, just recalibrate the thresholds.

## 4. `lidar_fps` mismatch debugging (PR B)

**Symptom.** `--duration 60` recorded ~20s on yesterday's pipeline.
Sensor warning said "depth fps 30 differs from config 10".

**First hypothesis.** FFT frequency axis is using the wrong fs → every
BPM reading wrong by 3× → yesterday's 10/10 and 12/12 "matches" are
coincidence.

**Verification.** Cross-checked Run #3's meta.json: `frames_seen / wall_seconds`
= 600 / 59.92 = **10.013 fps**. The pipeline loop genuinely runs at
config rate (sensor outputs at 30 fps but voxel/cluster/RR work
bottlenecks consumption to ~10). FFT axis used config rate. Today's
Run #3 BPM was internally consistent.

**Then re-checked yesterday's runs.** Same calculation: Run #1 was
7.61 fps, Run #2 was 3.95 fps. Yesterday the pipeline was slower
(pre-optimisation). FFT used config 10. So yesterday's BPMs WERE biased
by ratios. The 10/10 and 12/12 Garmin matches were coincidence after all.

**Fix.** Pipeline now measures wall-clock fps per RR window, passes
the measured value into `extract_rr_from_signal`. Demo gets a
`force_fps` override (synthetic data is sample-rate-coupled to config).

**Considered and rejected.** Treating the "10 vs 30" warning as a bug
to fix at sensor level. It's just a printout — the analysis is correct.
The deeper class-of-bug is "config drift between subsystems"; meta.json
now records both `fps_configured` and `fps_measured` for retrospective
audit.

## 6. Stale background failure mode

**Symptom.** First Polar-paired run today: residuals = total points
(~14k each frame), 66% of frames had NaN cz, but the pipeline STILL
reported "6.06 BPM, SNR 10.04, HIGH SETTLED" — confidently wrong.

**Root cause.** Synthetic background got written to
`data/background_model.npz` by an earlier `--mode demo` run (which set
`config.background_save_path` to the same file as live mode). Live
subtraction returned everything as foreground; subject selection
rejected the sprawling cluster as out-of-monitoring-volume; cz went
NaN for most frames.

**Why FFT still produced a number.** `extract_rr_from_signal`
mean-fills NaN. With 66% mean-fill, the remaining 34% of cz values
were sparse — but the FFT happily picked the strongest residual
periodic component, which happened to be at ~6 BPM (the temporal
structure of when the subject WAS detected, not respiration). The
high SNR was deceptive: the constant mean-fill DC-baseline plus a few
real samples produces a clean pseudo-peak.

**Fixed by.** (a) Recapture background with `bootstrap.sh --recapture-bg`.
(b) Patch `run_demo` to write to `background_model_demo.npz`, never
overwriting the live model.

**The bigger lesson.** Single-confidence-number is unreliable for
edge-case detection. Want an orthogonal signal-quality check: e.g.
"% of frames with NaN cz in the window". If that's > 20%, flag the
result regardless of SNR. Logged for tomorrow.

## 7. Apnoea detection lag (Run 5 breath-hold)

**Symptom.** Held breath for 56 s (manually marked rr=0 at start,
rr=15 at restart). Expected RR estimate to drop toward 0 during the
hold. Observed: BPM destabilised UPWARD (15 → 17 → 20 → 22 → 24)
across the hold and into recovery, with ~20 s lag before the drift
became obvious.

**What we expected.** A single signal: respiratory amplitude collapses
→ FFT power in band drops → RR confidence drops → "no breathing"
state in viewer.

**What actually happens.** When the dominant in-band peak vanishes,
the power-weighted centroid (PR I) loses its anchor and drifts to
whatever residual periodic components remain — the FFT doesn't return
zero, it returns garbage. Meanwhile peak-pick SNR stays moderate
because there's always SOMETHING that looks like a peak in the band.

**Implication.** Apnoea has **two signals**, not one:
- (a) chest amplitude cessation — visible in raw cz time series and in
  band-power magnitude
- (b) RR estimator instability — large window-to-window BPM swings,
  centroid drift

Either alone is unreliable. Combined, they're robust.

**Considered and rejected.** Single-signal apnoea detector ("if SNR
drops below X for Y seconds, flag apnoea"). Too easily fooled by
ordinary low-confidence windows. The dual-signal detector goes into
the architecture doc.

**Tomorrow.** PR J (sliding-window FFT) cuts the lag from ~20 s to
~10 s and makes signal (b) sharper. The dual-signal design is more
important than the FFT window choice.
